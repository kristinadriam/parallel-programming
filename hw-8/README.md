# lock-free множество

## состав

- `ordered_set.hpp` – 10.1 – lock-free множество (`lockfree::set<T>`).
- `linearizability_test.cpp` – 10.2 – нагрузочная проверка линеаризуемости&
- `negative_control.cpp` – контроль: доказывает, что харнесс умеет ловить нарушения.
- `Makefile` – сборка и запуск.

## сборка и запуск

```sh
make test    # линеаризуемость
make asan    # проверка памяти: use-after-free и утечки
make tsan    # проверка гонок данных
make neg     # негативный контроль
make run     # линеаризуемость + негативный контроль
```

параметры теста можно задавать вручную: `./lin_test <CONFIGS> <REPEATS>` (число случайных конфигураций и число конкурентных прогонов на конфигурацию).

## 10.1 – реализация

множество построено как lock-free отсортированный skip-list. skip-list дает поиск, вставку и удаление за O(log n).

### интерфейс

```cpp
template <typename T>
class set {
    bool add(T value);          // false – уже есть, true – добавлен
    bool remove(T value);       // false – не найден, true – удален
    bool contains(T value) const;
    bool isEmpty() const;
};
```

### гарантии прогресса

| метод | гарантия |
| ------- | ---------- |
| `add` | lock-free |
| `remove` | lock-free |
| `isEmpty` | lock-free |
| `contains` | lock-free; обход (спуск/вперед, без рестартов) wait-free для конечного домена, разовая регистрация участника EBR на поток – lock-free |

### ключевые идеи

- метка логического удаления – узлы выровнены, поэтому младший бит указателя `next` свободен и используется как флаг «узел логически удален». логическое удаление – это атомарная установка метки. физическое отцепление – последующий CAS.

- `find` спускается по уровням сверху вниз и на каждом уровне идет вперед, пока ключ соседа меньше искомого. помеченные (логически удаленные) узлы отцепляются попутно через CAS – helping, обеспечивающий lock-free прогресс.

- `add` вставляет узел случайной высоты: сначала CAS на уровне 0 – это точка линеаризации, затем достраивает верхние уровни. если узел успели удалить, достраивание прекращается.

- `remove` помечает уровни узла сверху вниз. пометка на уровне 0 – точка линеаризации и одновременно «коммит»: победитель CAS становится тем, кто удалил элемент (конкурент, увидев уже выставленную метку, возвращает `false`). после коммита узел полностью отцепляется и ретайрится.

- `contains` идет только вниз и вперед, без CAS и рестартов, и проверяет метку на уровне 0 – отсюда wait-free.

- `isEmpty` проходит уровень 0 и возвращает `false` на первом же непомеченном узле.

### безопасное освобождение памяти

применен Epoch-Based Reclamation (EBR).

## 10.2 – проверка линеаризуемости

1. берем случайное начальное состояние множества и случайный набор из `N` операций с операндами. аеребираем все `N!` последовательных порядков выполнения этих операций над эталонным `std::set`. для каждого порядка получаем результат каждой операции и записываем кортеж `[(операция_1:true/false), ..., (операция_N:true/false)]`. множество таких кортежей – это
все линеаризуемые исходы данного набора.

2. тот же набор операций запускаем в `N` потоков. собираем фактический кортеж результатов и проверяем, что он совпадает с одним из легальных кортежей шага 1. если нет – выполнение нелинеаризуемо.

каждая конфигурация повторяется многократно, чтобы поймать редкие чередования.

### негативный контроль

`negative_control.cpp` подставляет в тот же харнесс заведомо нелинеаризуемое множество. харнесс стабильно ловит нарушения – это доказывает, что тест действительно способен детектировать некорректность.

## результаты прогонов

- *линеаризуемость (18 000 конкурентных прогонов / 1 000 конфигураций)* – 0 нарушений.
- *AddressSanitizer (6 400 прогонов)* – нет use-after-free и утечек.
- *ThreadSanitizer (2 200 прогонов)*  – 0 предупреждений о гонках.
- *негативный контроль (100 прогонов)* – нарушения корректно пойманы.
